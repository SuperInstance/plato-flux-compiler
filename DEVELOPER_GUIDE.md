# Developer Guide — plato-flux-compiler

> Architecture deep-dive, module walkthrough, extension points, and contributing guide for the alarm DSL → FLUX bytecode compiler.

---

## Architecture Overview

`plato-flux-compiler` translates human-readable alarm condition strings into **FLUX bytecode** — a deterministic, portable instruction set executed by the `flux-core` VM. The same alarm logic runs identically on an ESP32 in a bilge pump and a GPU cluster in a data center.

The compilation pipeline is a classic front-end → middle → back-end:

```
Condition string
      │
      ▼
  Parser (recursive descent)
      │
      ▼
  AST (Condition tree)
      │
      ▼
  Codegen (tree walk → FLUX ops)
      │
      ▼
  Optimizer (peephole)
      │
      ▼
  FLUX bytecode (Vec<Op>)
```

### Why Bytecode?

The Plato system needs **deterministic** alarm evaluation across heterogeneous platforms. Source-level interpretation is slow and platform-dependent. Bytecode compilation gives:

- **Determinism:** Same bytecode → same result, everywhere
- **Performance:** Register-based VM with short-circuit evaluation
- **Portability:** flux-core runs on embedded, WASM, GPU, cloud
- **Debuggability:** Disassemble bytecode, inspect AST, unit test offline
- **Deployability:** Conditions compile at deploy time, not runtime

## Module-by-Module Walkthrough

### `parser` — Recursive-Descent Parser

Tokenizes and parses condition strings into an AST. The grammar:

```
condition     := or_expr
or_expr       := and_expr ("OR" and_expr)*
and_expr      := unary ("AND" unary)*
unary         := "NOT" unary | primary
primary       := comparison | range_check | threshold_check | "(" condition ")"
comparison    := IDENT operator NUMBER
operator      := ">" | ">=" | "<" | "<=" | "==" | "!="
range_check   := IDENT "in" "[" NUMBER "," NUMBER "]"
threshold_check := IDENT "rising_by" NUMBER "in" NUMBER "_ticks"
```

Precedence (low to high): OR < AND < NOT < primary.

**Parser API:**

```rust
let cond = Parser::parse("temp > 90 AND pressure < 100")?;
// Returns Condition AST node
```

**Error handling:** Returns `Result<Condition, ParseError>` with position information for invalid syntax.

**Extension point:** Add new primary expression types by:
1. Adding a new token type in the lexer
2. Adding a new match arm in `parse_primary()`
3. Creating a new `Condition` variant

### `ast` — Abstract Syntax Tree

The AST represents the parsed condition as a tree:

```rust
pub enum Condition {
    Comparison { sensor: String, operator: CmpOp, value: f64 },
    RangeCheck { sensor: String, low: f64, high: f64 },
    ThresholdCheck { sensor: String, delta: f64, ticks: u64 },
    And(Box<Condition>, Box<Condition>),
    Or(Box<Condition>, Box<Condition>),
    Not(Box<Condition>),
}
```

Key methods:
- `sensors()` — Collects all sensor names referenced in the condition
- `Display` — Pretty-prints the condition back to a string
- `PartialEq` — Structural equality for testing

**Extension point:** New `Condition` variants for future DSL features (e.g., `AverageCheck`, `RateOfChange`, `WindowedAggregation`).

### `codegen` — FLUX Bytecode Emitter

Walks the AST and emits FLUX bytecode (a `Vec<Op>` from flux-core). The codegen uses a register-based model:

- **R0** — Result register (0 = false, 1 = true)
- **R1–Rn** — Temporary registers for sensor values
- **Short-circuit evaluation:** AND jumps to false on left-false; OR jumps to true on left-true

**Codegen API:**

```rust
let mut codegen = Codegen::new();
codegen.compile(&condition)?;
let func = codegen.into_function("alarm_name".to_string());
// func.bytecode: Vec<Op>
// func.name: String
```

**Instruction patterns by node type:**

| Node | Generated Pattern |
|------|-------------------|
| `Comparison` | `LOAD R, [sensor]; PUSH val; CMP R, op` |
| `And` | Left + `JZ R0, skip` + Right + label |
| `Or` | Left + `JNZ R0, skip` + Right + label |
| `Not` | Inner + `INEG R0` |
| `RangeCheck` | `sensor >= low AND sensor <= high` (lowered) |
| `ThresholdCheck` | `sensor rising_by delta in N_ticks` (lowered to state tracking) |

**Disassembler:** `disassemble(&bytecode) → String` produces human-readable assembly for debugging.

**Extension point:** New codegen patterns for new AST nodes. Each node type maps to a private method on `Codegen`.

### `optimizer` — Peephole Optimizer

Post-codegen pass that improves generated bytecode:

1. **Dead code elimination:** Strips any instructions after `HALT`. If the program has already halted, subsequent code is unreachable.

2. **Jump threading:** Collapses chains of jumps. If `JZ target1` where `target1` is `JMP target2`, the first jump is patched directly to `target2`. This eliminates unnecessary indirection in complex boolean expressions.

**Optimizer API:**

```rust
let optimized = Optimizer::optimize(bytecode);
```

The optimizer is idempotent — running it twice produces the same output.

**Extension point:** Add new peephole rules by implementing pattern-matching functions on `&[Op]`. Common additions: constant folding, redundant load elimination, register allocation improvements.

### `integration` — Full Room Compilation

Compiles a complete room config (multiple alarms) into a single FLUX program with a dispatch table:

```rust
pub struct RoomConfig {
    pub room_name: String,
    pub room_type: String,
    pub alarms: Vec<AlarmDef>,
}

pub struct CompiledRoom {
    pub functions: Vec<CompiledFunction>,
    pub dispatch_table: Vec<(String, usize)>,  // (alarm_id, bytecode_offset)
    pub bytecode: Vec<Op>,  // Concatenated bytecode
}
```

The dispatch table maps alarm IDs to their entry points in the concatenated bytecode, enabling the flux-core VM to evaluate individual alarms without running the entire program.

**Integration API:**

```rust
let room = compile_room(&config)?;
```

- **Parser (12+ tests):** Each comparison operator, AND/OR/NOT, range checks, threshold checks, nested expressions, error handling for invalid syntax
- **AST (8+ tests):** Construction, display formatting, equality, sensor collection
- **Codegen (10+ tests):** Bytecode structure verification for each node type, disassembly output
- **Optimizer (5+ tests):** Dead code elimination, jump threading, idempotency
- **Integration (8+ tests):** Full room configs, fishing boat engine room, server rack, dispatch tables
- **Roundtrip (4+ tests):** Parse → compile → disassemble → verify structure

Run with:

```bash
cargo test
```

## Contributing Guide

### Adding a New DSL Operator

1. Add the token to the lexer in `parser.rs`
2. Add a new `Condition` variant in `ast.rs`
3. Add a parsing function in `parser.rs` (or extend `parse_primary()`)
4. Add a codegen method in `codegen.rs` for the new node
5. Add optimizer handling if the node generates jumps
6. Write tests: parser (valid + invalid), AST (display + equality), codegen (bytecode + disassembly), integration

### Adding a New Optimization

1. Identify the pattern to optimize in generated bytecode
2. Add a peephole rule in `optimizer.rs`
3. Ensure the rule preserves semantics (run before/after tests)
4. Add a test with bytecode that triggers the rule

### Code Style

- The parser should produce helpful error messages with position information
- All public types derive `Debug, Clone, PartialEq`
- Bytecode emission should be deterministic (same input → same output, always)
- Prefer lowering complex constructs to simpler ones in codegen (e.g., `RangeCheck` → `AND` of two comparisons)

## Design Decisions

| Decision | Rationale |
|----------|-----------|
| Recursive-descent parser | Simple, readable, easy to extend with new syntax |
| Register-based bytecode | Faster interpretation than stack-based; maps well to sensor reads |
| Short-circuit evaluation | AND/OR skip evaluation when result is determined — saves sensor reads |
| Dispatch tables | Evaluate individual alarms without running the full program |
| Peephole optimizer | Simple, effective, doesn't change semantics |
| Separate parser/codegen | Clean separation of concerns; testable independently |
